# HO-API 高并发架构优化 — 实施计划

## Context

当前系统在高并发场景下存在以下瓶颈：订单模块零缓存、无防缓存击穿/雪崩机制、无分布式锁（重复支付风险）、无定时任务（过期订单无清理、无统计预计算）、无全文搜索能力。本方案引入 L1 内存缓存 + 增强 Redis 缓存 + 分布式锁 + 幂等中间件 + Elasticsearch + 定时任务体系，全面提升系统吞吐和稳定性。

---

## Phase 1: 基础设施库（新建 4 个文件）

### 1.1 `src/lib/local-cache.ts` — L1 进程内 LRU 缓存
- Map-based LRU，TTL 自动过期
- 导出单例 `localCache`（maxSize=1000）
- 方法：`get<T>(key)`, `set(key, value, ttlMs)`, `del(key)`, `clear()`

### 1.2 增强 `src/lib/cache.ts` — 防击穿/雪崩/穿透
- 新增 `getOrLoad<T>(key, loader, ttl)` — Redis NX 互斥锁防击穿
- 新增 `jitteredTTL(base)` — ±20% 随机偏移防雪崩
- 空值缓存：loader 返回 null 时缓存 `{__empty:true}` 60 秒防穿透
- 保持现有 get/set/del/delByPattern 方法不变

### 1.3 `src/lib/distributed-lock.ts` — Redis 分布式锁
- `withLock<T>(key, fn, ttlMs=5000)` — NX+PX 获取锁，Lua 脚本安全释放
- 获取失败返回 null（调用方判断）
- Redis 不可用时降级放行（与 auth 熔断器同策略）

### 1.4 `src/lib/elasticsearch.ts` — ES 客户端
- `@elastic/elasticsearch` Client 单例
- `connectElasticsearch()` + `checkElasticsearchReadiness()`
- `ensureOrderIndex()` — 创建 orders 索引（mapping: userId, restaurantId, status, totalAmount, remark, createdAt）
- 所有操作静默失败，ES 不可用不阻断主业务

## Phase 2: 中间件（新建 1 个文件）

### 2.1 `src/middleware/idempotency.ts` — 幂等性中间件
- 读取 `X-Idempotency-Key` 请求头
- 命中 Redis `idempotent:<key>` 直接返回缓存响应
- 未命中则执行 handler 后缓存响应 1 小时
- 无 header 时透传不处理
- 应用于：order create/pay/cancel

---

## Phase 3: 订单模块优化（修改 4 个文件）

### 3.1 `src/modules/orders/order.repository.ts`
- `findById` / `findDetailById` 增加 Redis 缓存
  - completed/cancelled: TTL 3600s（终态不变）
  - paid: TTL 30s
  - pending: 不缓存
- `updateStatus` 时失效缓存 `cache:orders:id:<id>` + `cache:orders:detail:<id>`
- `create` 后同步写入 ES（静默失败）

### 3.2 `src/modules/orders/order.service.ts`
- `pay()` 包裹 `withLock('lock:order:pay:<id>', fn)`
- `cancel()` 包裹 `withLock('lock:order:cancel:<id>', fn)`
- 新增 `search()` 方法调用 ES 搜索

### 3.3 `src/modules/orders/order.route.ts`
- create/pay/cancel 应用 idempotency 中间件
- 新增 `POST /search` 端点（ES 全文搜索 + 多条件过滤）

### 3.4 `src/modules/orders/order.schema.ts`
- 新增 `searchOrderSchema`（keyword, status, restaurantId, dateFrom, dateTo, page, pageSize）

---

## Phase 4: 菜单模块优化（修改 1 个文件）

### 4.1 `src/modules/menu/menu.repository.ts`
- `findPaginated` / `findById` 加 L1 缓存层
  - 查询顺序：L1(5s) → L2 Redis(30s/300s) → DB
  - L1 命中直接返回，避免 Redis 网络往返

## Phase 5: 定时任务体系（新建 5 个文件）

### 5.1 `src/jobs/scheduler.ts` — 调度器
- `startScheduler()` 在 index.ts 启动时调用
- 每个 job 用 `setInterval` + 分布式锁（确保多 Pod 只执行一次）
- 注册所有 job 及其间隔

### 5.2 `src/jobs/order-cleanup.ts`
- 每 1 分钟：查找 pending 超过 30 分钟的订单，批量更新为 cancelled

### 5.3 `src/jobs/stats-calculator.ts`
- 每 5 分钟：按餐厅聚合当日订单数/营收/均价，写入 Redis Hash `stats:restaurant:<id>:<date>`
- 每天 00:05：汇总前一天数据写入 `daily_stats` 表

### 5.4 `src/jobs/cache-warmer.ts`
- 每 30 分钟：预加载活跃餐厅和热门菜单到 Redis 缓存

### 5.5 `src/jobs/order-sync.ts`
- 每 30 秒：增量同步新/更新的订单到 Elasticsearch
- 用 Redis key `es:orders:last_sync` 记录上次同步时间戳

## 验证方式

1. `bunx tsc --noEmit` — 类型检查通过
2. `bun run dev` — 开发环境启动成功（含 ES 容器）
3. 测试缓存：连续请求菜单列表，第 2 次应命中 L1 缓存（无 Redis 往返）
4. 测试分布式锁：并发 2 次 pay 同一订单，只有 1 次成功
5. 测试幂等：相同 X-Idempotency-Key 重复请求返回相同结果
6. 测试定时任务：等待 1 分钟确认过期订单被自动取消
7. 测试 ES 搜索：创建订单后约 30 秒可通过 search 端点搜索到
8. 测试统计：等待 5 分钟确认 Redis 中有 stats 数据，stats 接口可正常返回
