核心概念：ES 是"只读副本"

PostgreSQL（主数据）──同步──▶ Elasticsearch（搜索副本）
↑ 写入/更新                    ↑ 只用来查
│                              │
业务操作                      搜索/聚合/统计

你项目里的 order-sync.ts 就是这个模式——每 30 秒增量同步。这意味着：

- 写永远走 PostgreSQL
- 搜索走 ES（容忍 ≤30 秒延迟）
- ES 挂了，搜索不可用，但下单/付款/取消完全不受影响

总结一句话

ES 最适合的就是你说的——不变更或很少变更的历史数据，拿来做全文搜索和聚合分析。把它当成一个"可丢失、可重建的搜索索引"就
对了，不是缓存，也不是数据源。

关键前提是要建索引，裸查确实扛不住：

-- 不建索引：全表扫描，百万级就慢了
SELECT * FROM orders WHERE remark ILIKE '%关键词%';

-- 建 GIN 索引：百万级依然很快
ALTER TABLE orders ADD COLUMN remark_tsv tsvector
GENERATED ALWAYS AS (to_tsvector('simple', coalesce(remark, ''))) STORED;
CREATE INDEX idx_orders_remark_tsv ON orders USING GIN(remark_tsv);

-- 查询走索引
SELECT * FROM orders WHERE remark_tsv @@ to_tsquery('simple', '关键词');

最关键的 3 个差距

1. 消息队列（最重要）

Amazon 和阿里的交易链路都不是同步的：

你现在:   下单请求 → 验证 → 扣库存 → 写订单 → 写ES → 返回     (全同步，一个慢全部慢)

他们:     下单请求 → 验证 → 写订单 → 返回                     (只做核心操作)
↓ 发消息
┌─────────┼─────────┐
↓         ↓         ↓
扣库存    同步ES    发通知                     (异步消费，互不阻塞)

2. 分库分表

单 PostgreSQL 撑不过亿级订单。必须水平拆分：

orders 表 → 按 user_id % 64 分到 64 张表 → 分布在 8 个 PG 实例

Amazon 用 DynamoDB 天然分片，阿里用 DRDS + OceanBase。

3. 多级降级体系

不只是"ES 挂了搜索不可用"这么简单，而是：

全部正常 → 正常返回个性化搜索结果
ES 慢    → 返回 Redis 缓存的热门商品
Redis 慢  → 返回本地缓存的兜底列表
全挂      → 返回静态 CDN 页面（至少能看到东西）

务实建议

你的项目当前架构在 日订单 10 万级以内完全够用。真正需要升级时，优先级是：

1. 加消息队列 (RocketMQ/Kafka)  ← 性价比最高，解耦 + 削峰 + 异步
2. Redis Cluster                ← 缓存层水平扩展
3. 数据库分库分表或换 TiDB       ← 数据量瓶颈时
4. ES 集群化                    ← 搜索量大时
5. 链路追踪 (OpenTelemetry)     ← 排查问题用

不需要一步到位，Amazon 2003 年也是单体 + Oracle，阿里 2009 年双十一也才刚从 Oracle 迁到
MySQL。架构是随业务量演进的，不是一开始就设计到终态。

结论

推荐 Kafka，理由：

1. Node.js/Bun 生态：kafkajs 是纯 JS 实现，零原生依赖，Bun 兼容性好；RocketMQ 的 Node 客户端要走 gRPC，依赖重且社区小
2. KRaft 模式：Kafka 3.5+ 已移除 ZooKeeper 依赖，单进程即可运行，运维复杂度大幅降低
3. 延迟消息替代：你已有 Redis + scheduler 做定时关单，不依赖 MQ 的延迟消息
4. 事务消息替代：用本地事务表 (outbox pattern) + Kafka 可靠投递，比 RocketMQ 半消息更通用
5. 社区与生态：Kafka 全球用户基数远大于 RocketMQ，排错资料多，云服务商全线支持

如果你的团队是纯阿里技术栈且用 Java，选 RocketMQ。但 Bun + TypeScript 生态下 Kafka 是更务实的选择。
