前置条件：

本步目标（只做第四步）：

请完成以下工作：

1. 新增库存应用服务：
- 新建 `src/modules/orders/inventory.service.ts`
- 实现函数：
    - `reserveStock(skuId, qty, orderId, idempotencyKey)`
        - Redis: `DECRBY stock:{skuId} qty`（结果 < 0 时立即回补并返回失败）
        - 成功后写 `stock_ledger(reason=reserve, delta=-qty)`
        - 写 `outbox_events(event_type=inventory.decrement.requested, status=pending)`
    - `commitStockToDb(skuId, qty, orderId, idempotencyKey)`
        - 调用 `decrementSkuStockInDb`
        - 成功写 `stock_ledger(reason=commit)`
        - 失败写 `outbox_events(...failed...)` 或标记失败待重试
    - `rollbackReserveStock(skuId, qty, orderId, idempotencyKey)`
        - Redis 回补 `INCRBY`
        - 写 `stock_ledger(reason=rollback, delta=+qty)`

2. 增加 Outbox 处理入口（非真实 MQ）：
- 在 `src/modules/orders/outbox.worker.ts` 增加 `processPendingOutboxBatch(limit=100)`
- 流程：
    - 拉取 pending/可重试事件
    - 针对 `inventory.decrement.requested` 调用 `commitStockToDb`
    - 成功 `markOutboxSent`
    - 失败 `markOutboxFailed` 并设置 `next_retry_at`（简单退避：`now + retry_count*30s`）

3. 在订单模块接入最小调用点：
- 修改 `src/modules/orders/order.service.ts`（或现有下单入口）
- 下单时调用 `reserveStock(...)`
- 若预扣失败，返回库存不足错误
- 订单创建成功后仅保证 outbox 已写入；不要求同步完成 DB 扣减

4. 增加一个手动触发路由（仅开发调试）：
- 如 `POST /internal/outbox/process`
- 调用 `processPendingOutboxBatch`
- 返回处理数量与成功/失败统计

5. 幂等与一致性要求：
- 同一 `idempotency_key` 重复请求不得重复记账
- Redis 成功但 DB 失败时，可通过 outbox 重试恢复
- 明确记录日志（成功、失败、补偿）便于排障

6. 验证：
- 提供最小联调脚本或步骤（可用 `apitest/*.http`）验证：
    - 正常下单：Redis 预扣成功 + ledger/outbox 写入
    - 重复请求：幂等生效
    - 模拟 DB 失败：事件进入 failed/pending retry
    - 手动触发 outbox：状态变 sent 或 retry_count 增长
- 执行后输出：
    - 变更摘要
    - 修改文件列表
    - 验证结果（命令/请求 + 关键输出）
    - 剩余 TODO（真实 MQ、分布式锁、监控告警）

约束：
- 只做第四步，不扩展到完整消息中间件接入
