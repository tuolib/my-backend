阶段二：核心服务设计 - API 网关

⏺ API 网关模块设计

输入接口

- HTTP入口: 接收所有客户端请求 (REST JSON)
- 请求头: Authorization: Bearer <JWT>, X-Request-ID, X-Client-Version
- 限流参数: 基于 IP + UserID 的令牌桶 (Redis DECR)

输出接口

- 代理转发: 根据路由规则分发至下游微服务 (用户/商品/订单/支付/营销/搜索)
- 统一响应格式: { code, data, message, requestId }
- 错误响应: 401/403/429/502/504 标准化错误体

依赖服务

┌──────────┬─────────────────────────────────────────┐
│   依赖   │                  用途                   │
├──────────┼─────────────────────────────────────────┤
│ Redis    │ JWT黑名单校验、令牌桶限流计数、会话缓存 │
├──────────┼─────────────────────────────────────────┤
│ 各微服务 │ HTTP代理转发 (服务发现/静态配置)        │
└──────────┴─────────────────────────────────────────┘

健壮性与容错

- 熔断: 下游服务连续失败 N 次后熔断,返回降级响应,定期半开探测恢复
- 超时控制: 每个下游服务独立超时配置 (默认 3s, 支付 10s)
- 限流: 全局 + 用户级双层令牌桶,突发流量返回 429
- 重试: 仅对 GET 幂等请求重试 1 次,非幂等请求不重试
- 健康检查: /health 端点,含 Redis 连通性探测
- 优雅关闭: 收到 SIGTERM 后停止接收新请求,等待在途请求完成 (最长 30s)
