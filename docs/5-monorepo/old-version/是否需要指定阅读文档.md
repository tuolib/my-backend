

# 1 我要把架构路线图的文件放在项目中，再加比如 stage1 的纲目图放在项目中让你阅读吗？

不需要那么麻烦。让我给你设计一个更轻量的方案：

---

## 两种方式对比

| 方式 | 操作 | Token 消耗 | 推荐 |
|------|------|-----------|------|
| ❌ 上传文件让 AI 读 | 每轮都要读文件，重复消耗 | 高 | 否 |
| ✅ 对话开头带一段"项目简历" | 复制粘贴几行文字 | 极低 | **是** |

## 你只需要做一件事

每次**开新对话**时，在提示词最前面粘贴一段**项目简历**（约 10-15 行），然后直接提需求。这比上传文件更快、更省 token。

### 项目简历模板（存在你本地备忘录里，随项目进展更新）

```
## 项目简历
- 项目：企业级高并发电商系统
- 技术栈：Bun + Hono + Drizzle ORM + PostgreSQL + Redis + Docker + Caddy
- 结构：DDD 分层 src/{domain}/{controller,service,repository,schema,types}
- 公共层：src/shared/{middleware,utils,config,types}
- 统一响应：ApiResponse<T> { success, data, error, meta }
- 错误处理：AppError 类 + errorHandler 中间件
- 认证：JWT authGuard + roleGuard
- 缓存：redis 实例（ioredis），Cache-Aside 模式
- 数据库：Drizzle ORM，BaseRepository<T> 泛型基类

## 已完成
- ✅ 阶段 1：工程骨架
- ✅ 阶段 2：数据库建模
- 🔄 阶段 3：商品域（进行中，已完成 types.ts / schema.ts）

## 当前任务
阶段 3 → Step 2 → 生成 repository.ts
```

### 实际使用时就是这样

```
{粘贴项目简历}

---
请生成 src/domain/product/repository.ts

已有导出：
- types.ts: Product, Sku, CreateProductInput, ProductFilter
- schema.ts: createProductSchema, productQuerySchema

要求：继承 BaseRepository，支持游标分页+多条件过滤。只输出代码。
```

## 项目简历的更新节奏

每完成一个阶段，花 30 秒更新两个地方就行：

1. **「已完成」列表**打个 ✅
2. **「公共层」描述**加上新增的共享模块（如果有的话）

比如完成阶段 4 后加一行：
```
- 认证：JWT authGuard + roleGuard（阶段 4 新增）
```

这样做的好处是 AI 每次都能精准知道你的项目现状，又不浪费 token 去读文件。路线图文件留在本地给你自己看进度就好，不用喂给 AI。



# 2 那你之前告诉我的方式和这个不一样

你说得对，我之前给的两套方案确实有冲突。让我理清楚，给你一个**统一的、明确的操作流程**：

---

## 结论：两套方案结合用，各管各的阶段

```
阶段提示词（路线图里那个大 prompt）
  → 用于 Step 1，一次性生成该阶段的完整骨架代码

4 步法（要清单 → 逐文件 → 组装 → 修复）
  → 用于 Step 1 产出不完整或需要深入补充的情况
```

## 实际操作流程（最终版）

### 每个阶段你只需要做 3 步：

```
┌─────────────────────────────────────────────────────┐
│  Step A  🚀 用阶段提示词一次生成                      │
│  直接复制路线图里的大 prompt 发给 AI                   │
│  拿到该阶段 80% 的代码                                │
├─────────────────────────────────────────────────────┤
│  Step B  🔧 检查 + 补漏                              │
│  本地跑一下，缺什么文件或报错就针对性补问              │
│  用「项目简历 + 精准问题」模式                        │
├─────────────────────────────────────────────────────┤
│  Step C  📝 更新项目简历                              │
│  花 30 秒更新已完成列表和新增的共享模块                │
│  为下一个阶段做准备                                   │
└─────────────────────────────────────────────────────┘
```

### 具体怎么操作：

**Step A — 阶段 1 为例**

直接发路线图里的提示词：
```
你是一位资深后端架构师。请帮我用 Bun + Hono 搭建一个企业级电商项目的基础骨架。
要求：
1. 目录结构采用 DDD 分层...
（完整复制路线图里的 prompt）
```
→ AI 一次输出所有文件，你直接拿去用

**Step B — 如果一次没生成完整，或者跑起来有问题**

这时候切换到精准补问模式：
```
{项目简历}

---
阶段 1 已生成以下文件：
- src/shared/config/env.ts ✅
- src/shared/middleware/error-handler.ts ✅
- src/shared/middleware/logger.ts ✅
- src/shared/types/response.ts ✅
- src/index.ts ✅
- Dockerfile ✅
- docker-compose.yml ✅
- Caddyfile ❌ 缺失

请只生成 Caddyfile，要求反向代理到 app:3000，自动 HTTPS。
```

或者修 bug：
```
{项目简历}

---
运行 bun run dev 报错：
Cannot find module 'src/shared/config/env' from 'src/index.ts'

请修复导入路径问题，只输出需要改的行。
```

**Step C — 更新项目简历**
```
## 已完成
- ✅ 阶段 1：工程骨架
  - 共享层：ApiResponse, AppError, errorHandler, requestLogger, env config
  - 基础设施：Dockerfile, docker-compose, Caddyfile
```

### 什么时候用「4 步法」里的逐文件生成？

只有当某个阶段**特别复杂**，一次 prompt 生成的代码质量不够好时才用。主要是：

| 阶段 | 推荐方式 |
|------|---------|
| 阶段 1-5 | 直接用阶段提示词一次生成 |
| **阶段 6（订单）** | 4 步法逐文件，因为涉及状态机+幂等+事务太复杂 |
| **阶段 7（库存）** | 4 步法逐文件，Lua 脚本+三态流程需要精细控制 |
| 阶段 8-10 | 直接用阶段提示词一次生成 |

---

所以简单来说：**大 prompt 是主力，4 步法是补刀工具，项目简历是跨阶段的记忆纽带**。你现在可以直接复制阶段 1 的提示词发给我开始了。
