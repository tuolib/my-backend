关键设计决策

Swarm 集群：3 Manager + 2 Worker
- 3 个 Manager 满足 Raft 共识的最小奇数要求，挂 1 台仍有多数派
- S1/S2 虽然跑数据库，兼任 Manager 没问题（<1000 QPS 负载不高）

PostgreSQL 高可用：Streaming Replication + Patroni
- S1 Primary → S2 Replica，同步流复制
- Patroni 管理自动故障切换，Primary 挂了 Replica 自动提升
- Patroni 用 etcd 做共识（部署在 S1/S2/S3 上，3 节点）

Redis 高可用：Sentinel 模式
- S1 Master → S2 Replica
- 3 个 Sentinel 分布在 S1/S2/S3，自动完成故障切换
- 应用通过 Sentinel 发现当前 Master，不硬编码地址

应用服务：Swarm 调度到 S4/S5
- 每个服务 replicas=2，Swarm 分散在两个 Worker 上
- 任一 Worker 挂了，Swarm 自动把副本调度到另一台
- 内置 rolling update，零停机发布

流量路径

用户 → DNS → S3 Caddy → Swarm Ingress Mesh
↓
S4/S5 上的 API Gateway (轮询)
↓
S4/S5 上的各微服务 (轮询)
↓
S1/S2 PG + Redis (通过 Patroni/Sentinel 自动路由)

这个方案在 <1000 QPS 下足够用

- 数据层无单点（PG + Redis 都有自动故障切换）
- 应用层无单点（每个服务 2 副本）
- Swarm 控制面无单点（3 Manager）
- 后续 QPS 增长，先加 Worker 节点水平扩容应用副本即可

基础设施层（约束到 Manager 节点）：

- postgres-primary：约束到 node.labels.role == db-primary（S1）
    - image: postgres:16-alpine
    - 挂载 postgresql.conf + init.sql
    - volume: pg_primary_data
    - healthcheck: pg_isready
- postgres-replica：约束到 node.labels.role == db-replica（S2）
    - image: postgres:16-alpine
    - 以 streaming replication 模式启动
    - volume: pg_replica_data
    - 环境变量指向 primary
- redis-primary：约束到 node.labels.role == db-primary（S1）
    - image: redis:7-alpine
    - 挂载 redis.conf
    - volume: redis_primary_data
- redis-replica：约束到 node.labels.role == db-replica（S2）
    - image: redis:7-alpine
    - command: redis-server --replicaof redis-primary 6379
- redis-sentinel（replicas=3）：约束到 Manager 节点（S1/S2/S3 各一个）
    - image: redis:7-alpine
    - command: redis-sentinel
    - 配置监控 redis-primary
- caddy：约束到 node.labels.role == gateway（S3）

- placement: node.role == worker

- healthcheck: POST /health

架构要点

docker-stack.yml 包含 12 个服务：
- postgres-primary → S1 (node.labels.role == db-primary)
- postgres-replica → S2 (node.labels.role == db-replica)，通过 pg_basebackup -R 自动初始化流复制
- redis-primary → S1，redis-replica → S2（--replicaof 主从）
- redis-sentinel x3 → 分布在 3 个 Manager 节点（S1/S2/S3）
- caddy → S3 (node.labels.role == gateway)，host mode 直绑 80/443
- 5 个应用服务 x2 副本 → Worker 节点（S4/S5），start-first 零停机更新

安全设计：
- 敏感值（JWT secrets、PG 密码）通过 Docker Secrets 注入，运行时从 /run/secrets/ 读取
- DATABASE_URL 在容器启动时由 sh -c 动态拼接，密码不出现在 stack 文件中
- 应用服务限制 CPU 0.5 核 + 512MB 内存

deploy.sh 支持的命令：
- ./deploy.sh init — 初始化 Swarm + 输出 join token
- ./deploy.sh labels — 交互式设置节点标签
- ./deploy.sh secrets — 创建 Docker Secrets（含长度校验）
- ./deploy.sh build — 构建并推送所有服务镜像
- ./deploy.sh deploy — 部署 Stack（含前置检查）
- ./deploy.sh full — 完整流程一键执行
