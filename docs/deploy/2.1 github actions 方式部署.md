# Plan: GitHub Actions CI/CD — 提交代码自动部署到 Swarm

## Context

Swarm 集群和手动部署脚本（`infra/swarm/deploy.sh`）已完成。现在需要实现：**推送代码到 GitHub main 分支 → 自动构建 5 个微服务镜像 → 自动部署到 Swarm 集群**。

现有 `.github/workflows/deploy.yml` 是旧的单体架构工作流（构建单个 `ho-api` 镜像），需要替换为新的微服务工作流。

## 要做的事

### 1. 替换 `.github/workflows/deploy.yml`

**整个文件替换**，新工作流包含 4 个 Job：

```
push to main ──→ ① 类型检查 ──→ ② 并行构建 5 个镜像 ──→ ③ SSH 部署到 Swarm ──→ ④ 健康检查
PR to main   ──→ ① 类型检查 ──→ ② 并行构建（不推送，不部署）
手动触发     ──→ ① 类型检查 ──→ ② 并行构建 5 个镜像 ──→ ③ SSH 部署到 Swarm ──→ ④ 健康检查
```

**Job ① `check`** — 类型检查
- `bun install --frozen-lockfile` + `bunx tsc --noEmit`

**Job ② `build-images`** — Matrix 并行构建 5 个镜像
- 5 个服务并行构建，推送到 GHCR（`ghcr.io/<owner>/ecom-<service>:<sha12>`）
- PR 时只构建不推送
- 每个服务独立 GHA 缓存（`scope=${{ matrix.service.name }}`）
- `fail-fast: true`：任何一个构建失败，全部取消

**Job ③ `deploy`** — SSH 到 Swarm Manager 部署
- 用 `appleboy/ssh-action` SSH 到 S1
- 登录 GHCR，逐个执行 `docker service update --with-registry-auth --image <新镜像> ecom_<服务>`
- 每个服务更新后等待收敛，检测 `completed` 或 `rollback_completed` 状态
- 顺序更新（非并行），限制爆炸半径

**Job ④ `smoke-test`** — 外部健康检查
- 从 GitHub Runner 访问 `POST https://api.find345.site/health`
- 最多重试 20 次，每次间隔 10 秒
- 输出部署摘要到 GitHub Step Summary

### 2. 不需要修改的文件

- 所有 Dockerfile — 已适配生产
- `infra/swarm/docker-stack.yml` — 镜像命名格式已兼容
- `infra/swarm/deploy.sh` — 仅用于首次手动部署
- 应用代码 — 无需改动

## 部署方式：SSH（而非 Self-hosted Runner）

选择 SSH 而非 self-hosted runner，原因：
- 不需要在服务器上安装 GitHub Runner 软件
- 只需一个 SSH 密钥即可
- `--with-registry-auth` 自动将 GHCR 凭证转发给 Worker 节点

## 需要配置的 GitHub Secrets

| Secret 名称 | 值 | 说明 |
|---|---|---|
| `SWARM_HOST` | S1 的公网 IP | SSH 目标 |
| `SWARM_USER` | `root` 或 `deploy` | SSH 用户名 |
| `SWARM_SSH_KEY` | SSH 私钥内容 | SSH 认证 |
| `GHCR_PAT` | GitHub PAT（`read:packages` + `write:packages`）| Swarm Manager 拉取镜像用 |

## 服务器端 SSH 准备（用户手动做）

1. 生成 SSH 密钥对：`ssh-keygen -t ed25519 -C "github-actions" -f ~/.ssh/github_deploy`
2. 公钥加到 S1 的 `~/.ssh/authorized_keys`
3. 私钥内容粘贴到 GitHub Secret `SWARM_SSH_KEY`

## 回滚机制

- **自动回滚**：`docker-stack.yml` 已配置 `failure_action: rollback`，健康检查失败时 Swarm 自动回滚
- **手动回滚**：`docker service update --rollback ecom_<service>` 或指定旧 tag 重新 update

## 验证

1. 提交 PR 到 main → 检查类型检查 + 构建通过（不部署）
2. 合并到 main → 检查完整流水线：构建 → 推送 → 部署 → 健康检查
3. 手动触发 workflow_dispatch → 自定义 tag 部署
4. 故意提交一个会崩溃的代码 → 验证 Swarm 自动回滚

# 选择 SSH 而非 self-hosted runner，这个原因再帮忙解释一下，我没懂

⏺ 两种方式的核心区别在于 谁来执行部署命令：
                                         
---                                                                                                                                  
Self-hosted Runner（旧方案）

GitHub Actions ──→ 服务器上常驻的 Runner 进程 ──→ 直接执行 docker 命令

你需要在 Swarm Manager（S1）上安装并运行一个 GitHub Actions Runner 守护进程。它长期驻留在服务器上，监听 GitHub
的任务分发，收到任务后在本地执行。

缺点：
- 要在服务器上安装 Runner 软件、配置自启动、保持更新
- Runner 进程本身占用资源
- Runner 挂了或掉线，部署就卡住
- 旧工作流的 runs-on: [self-hosted, linux, swarm-manager] 就是这个方案

SSH（新方案）

GitHub Actions Runner (GitHub 云端) ──SSH──→ 服务器 ──→ 执行 docker 命令

部署 Job 跑在 GitHub 提供的云端 Runner 上（runs-on: ubuntu-latest），通过 SSH 远程连接到 S1 执行命令。服务器上
不需要安装任何额外软件，只要 SSH 能连上就行。

优点：
- 服务器零维护 — 不装 Runner、不管进程、不担心掉线
- 只需配一个 SSH 密钥（已有的 authorized_keys 机制）
- --with-registry-auth 会把 GHCR 登录凭证自动转发给 Worker 节点拉镜像，不需要在每台 Worker 上单独登录

简单说：SSH 方案把服务器当纯粹的执行目标，所有调度逻辑都在 GitHub 云端完成，服务器不需要知道 GitHub Actions 的存在。







❯ 原来已经有的 self-hosted runner 如何停止？

⏺ SSH 到 S1 服务器后，按以下步骤操作：

1. 找到 Runner 安装目录

# 通常在这些位置之一
ls /home/*/actions-runner/ 2>/dev/null                    
ls /opt/actions-runner/ 2>/dev/null

# 或者通过进程找
ps aux | grep actions.runner

2. 停止服务

cd /path/to/actions-runner

# 如果注册为 systemd 服务（最常见）
sudo ./svc.sh stop
sudo ./svc.sh uninstall

# 如果是前台运行的，直接 kill 进程即可

3. 从 GitHub 注销

# 在 Runner 目录下执行
./config.sh remove --token <REMOVE_TOKEN>

REMOVE_TOKEN 的获取方式：GitHub 仓库 → Settings → Actions → Runners → 点击该 Runner → 页面上会有 Remove 按钮和对应的 token。

或者更简单：直接在 GitHub 网页上删除 — 同一个页面点 Remove 按钮即可，不需要在服务器上执行 config.sh remove。

4. 清理文件（可选）

rm -rf /path/to/actions-runner

  ---
总结就三步：停服务 → 从 GitHub 注销 → 删文件。注销后 GitHub 就不会再给这台机器派发任务了。





# github actions 方式部署，需要设置 docker node 节点名称吗？如何执行命令行？

⏺ 不需要。GitHub Actions 工作流里的部署命令不涉及节点名称。

原因

工作流里用的是 docker service update，它只告诉 Swarm "把这个服务的镜像换成新版本"，至于容器跑在哪个节点上，是 Swarm 调度器根据       
docker-stack.yml 里的 placement.constraints 自动决定的：

# docker-stack.yml 中已配好
deploy:
placement:
constraints:
- node.role == worker    # 应用服务跑在 Worker 节点
- node.labels.role == db-primary  # 数据库跑在数据节点

但是节点 label 需要提前设好

这是在首次部署时（deploy.sh）做的事，只需要做一次：

# 在 Swarm Manager (S1) 上执行
docker node update --label-add role=db-primary  S1
docker node update --label-add role=db-replica  S2
docker node update --label-add role=gateway     S3
# S4/S5 是 Worker，默认 node.role == worker，不需要额外 label

可以验证：

docker node ls -q | xargs -I{} docker node inspect {} --format '{{ .Description.Hostname }}: {{ .Spec.Labels }}'

总结

┌───────────────────────────────────┬──────────────────┬──────────────────┐
│               事项                │      谁负责      │     什么时候     │
├───────────────────────────────────┼──────────────────┼──────────────────┤
│ 节点 label 设置                   │ deploy.sh / 手动 │ 首次部署，一次性 │
├───────────────────────────────────┼──────────────────┼──────────────────┤
│ 服务调度到哪个节点                │ Swarm 调度器     │ 自动，每次部署   │
├───────────────────────────────────┼──────────────────┼──────────────────┤
│ GitHub Actions 需要知道节点信息吗 │ 不需要           │ —                │
└───────────────────────────────────┴──────────────────┴──────────────────┘

GitHub Actions 只管 "更新镜像"，Swarm 自己决定 "在哪里运行"。

⏺ 下面是首次部署的完整操作过程：

首次部署：5 步走

全部在 S1（Swarm Manager） 上执行。可以一步步来，也可以用 ./deploy.sh full 一次走完。
                                                                                                                                       
---                                                                                                                                
第 1 步：初始化 Swarm 集群

# SSH 到 S1
./deploy.sh init

做了什么：
- 在 S1 上执行 docker swarm init，把 S1 变成 Swarm Manager
- 输出两条 docker swarm join 命令（一条给 Manager，一条给 Worker）

你要做的：
- 复制 Manager join 命令，SSH 到 S2、S3 分别执行（它们作为 Manager 节点）
- 复制 Worker join 命令，SSH 到 S4、S5 分别执行（它们作为 Worker 节点）

执行完后可以在 S1 上验证：
docker node ls
# 应该看到 5 个节点，3 个 Manager + 2 个 Worker

  ---
第 2 步：设置节点标签

./deploy.sh labels

做了什么：
- 列出所有节点，交互式询问每个节点的角色
- 你根据提示输入：

┌──────┬────────────┬──────────────────────────┐
│ 节点 │    输入    │           作用           │
├──────┼────────────┼──────────────────────────┤
│ S1   │ db-primary │ PostgreSQL 主 + Redis 主 │
├──────┼────────────┼──────────────────────────┤
│ S2   │ db-replica │ PostgreSQL 从 + Redis 从 │
├──────┼────────────┼──────────────────────────┤
│ S3   │ gateway    │ Caddy 反向代理           │
├──────┼────────────┼──────────────────────────┤
│ S4   │ skip       │ Worker，跑应用服务       │
├──────┼────────────┼──────────────────────────┤
│ S5   │ skip       │ Worker，跑应用服务       │
└──────┴────────────┴──────────────────────────┘

Swarm 根据这些标签决定每个容器调度到哪台机器。

  ---
第 3 步：创建 Docker Secrets

./deploy.sh secrets

做了什么：
- 交互式要求你输入 4 个密钥值（输入时不显示字符）：

┌────────────────────┬────────────────────┬──────────────────┐
│       Secret       │        说明        │       建议       │
├────────────────────┼────────────────────┼──────────────────┤
│ postgres_password  │ 数据库密码         │ 随机强密码       │
├────────────────────┼────────────────────┼──────────────────┤
│ jwt_access_secret  │ JWT 签名密钥       │ 至少 16 位随机串 │
├────────────────────┼────────────────────┼──────────────────┤
│ jwt_refresh_secret │ JWT 刷新密钥       │ 至少 16 位随机串 │
├────────────────────┼────────────────────┼──────────────────┤
│ internal_secret    │ 服务间内部通信密钥 │ 至少 8 位随机串  │
└────────────────────┴────────────────────┴──────────────────┘

这些密钥存储在 Swarm 的加密 Raft 日志中，运行时以文件形式挂载到容器的 /run/secrets/ 下。

  ---
第 4 步：构建并推送镜像

REGISTRY=ghcr.io/<你的用户名> TAG=v1.0.0 ./deploy.sh build

做了什么：
- 依次构建 5 个服务的 Docker 镜像
- 推送到你指定的 Registry

前提： 先登录 Registry：
echo "<你的GHCR_PAT>" | docker login ghcr.io -u <你的用户名> --password-stdin

注意：这一步以后由 GitHub Actions 自动完成，首次手动做一次就行。

  ---
第 5 步：部署 Stack

REGISTRY=ghcr.io/<你的用户名> TAG=v1.0.0 ./deploy.sh deploy

做了什么：
- 检查 Swarm 状态、Secrets 是否存在、节点标签是否齐全
- 执行 docker stack deploy -c docker-stack.yml ecom
- Swarm 根据配置文件自动把容器分发到各节点

  ---
完整流程一键执行

如果不想分步，可以直接：

REGISTRY=ghcr.io/<你的用户名> TAG=v1.0.0 ./deploy.sh full

它会按顺序执行 init → labels → secrets → build → deploy，中间有暂停让你去其他节点执行 join 命令。

首次部署完成后，后续每次推送代码到 main 分支，GitHub Actions 会自动接管：构建新镜像 → SSH 到 S1 → docker service update
滚动更新，不再需要手动操作。
