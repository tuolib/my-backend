# Patroni 迁移操作手册（新手向）

> 适用场景：已有 `postgres-primary` / `postgres-replica` 在运行，迁移到 Patroni 自动故障转移架构。

---

## 总览：一共 9 步

```
步骤 1. 备份数据库          ← 安全网，出问题可以恢复
步骤 2. 创建复制用户        ← Patroni 需要一个专门的复制账号
步骤 3. 创建 Docker Secret  ← 把复制密码安全地存到 Swarm
步骤 4. 更新节点标签        ← db-primary → db-1，db-replica → db-2
步骤 5. 构建 Patroni 镜像   ← 通过 CI 或手动构建
步骤 6. 上传配置文件        ← 把新的配置文件传到服务器
步骤 7. 部署新 Stack        ← 核心操作，会自动替换旧服务
步骤 8. 验证集群状态        ← 确认一切正常
步骤 9. 测试故障转移        ← 模拟宕机，看是否自动切换
```

预计总耗时：20-30 分钟（其中停机 2-5 分钟）

---

## 步骤 1：备份数据库

> **在哪执行：** 任意一台 Swarm Manager 节点（通常是 S1）

先找到正在运行的 postgres-primary 容器：

```bash
docker ps -f name=ecom_postgres-primary
```

你会看到类似这样的输出：

```
CONTAINER ID   IMAGE               STATUS        NAMES
a1b2c3d4e5f6   postgres:16-alpine  Up 2 days     ecom_postgres-primary.1.xxx
```

然后执行备份：

```bash
# 导出整个数据库到当前目录
docker exec $(docker ps -q -f name=ecom_postgres-primary) \
  pg_dump -U postgres ecommerce > backup_$(date +%Y%m%d).sql
```

**确认成功：**

```bash
# 检查备份文件大小，不应该是 0
ls -lh backup_*.sql
```

如果文件大小合理（比如几 MB 或更大），备份成功。

---

## 步骤 2：创建复制用户

> **在哪执行：** 同一台 Manager 节点（S1）

Patroni 需要一个叫 `replicator` 的专用账号来做主从复制。先想一个**强密码**，记下来，后面步骤 3 还要用。

```bash
# 把 <你的复制密码> 替换成你想好的密码
docker exec $(docker ps -q -f name=ecom_postgres-primary) \
  psql -U postgres -c "CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD '<你的复制密码>';"
```

**确认成功：** 输出 `CREATE ROLE` 就对了。

验证用户存在：

```bash
docker exec $(docker ps -q -f name=ecom_postgres-primary) \
  psql -U postgres -c "\du replicator"
```

应该看到 `replicator` 用户，属性里有 `Replication`。

---

## 步骤 3：创建 Docker Secret

> **在哪执行：** 任意 Manager 节点

Docker Secret 是 Swarm 的密码管理机制，密码加密存储，只有需要的容器才能读取。

```bash
# 把 <你的复制密码> 替换成步骤 2 中设置的同一个密码
echo -n "<你的复制密码>" | docker secret create replication_password -
```

> **注意：** `echo -n` 的 `-n` 很重要，不加会多一个换行符导致密码不匹配。

**确认成功：**

```bash
# 列出所有 Secret，应该能看到 replication_password
docker secret ls
```

输出中应有：

```
ID            NAME                   CREATED
xxxx          postgres_password      3 weeks ago
xxxx          replication_password   1 second ago    ← 新的
xxxx          jwt_access_secret      3 weeks ago
...
```

---

## 步骤 4：更新节点标签

> **在哪执行：** 任意 Manager 节点

以前标签叫 `db-primary` / `db-replica`（暗示固定角色），现在改成 `db-1` / `db-2`（只代表节点编号，角色由 Patroni 动态决定）。

先看看当前节点和标签：

```bash
docker node ls
```

记住 S1 和 S2 的 **HOSTNAME**，然后执行（替换 `<S1主机名>` 和 `<S2主机名>`）：

```bash
# 删除旧标签
docker node update --label-rm role <S1主机名>
docker node update --label-rm role <S2主机名>

# 添加新标签
docker node update --label-add role=db-1 <S1主机名>
docker node update --label-add role=db-2 <S2主机名>
```

> S3（gateway）的标签不需要改。

**确认成功：**

```bash
# 查看所有节点的标签
docker node ls -q | xargs -I{} sh -c \
  'echo "$(docker node inspect --format "{{.Description.Hostname}}" {}) → $(docker node inspect --format "{{range \$k,\$v := .Spec.Labels}}{{\$k}}={{\$v}} {{end}}" {})"'
```

应该看到：

```
S1主机名 → role=db-1
S2主机名 → role=db-2
S3主机名 → role=gateway
```

## 步骤 5：构建 Patroni 镜像

有两种方式，选一种即可。

### 方式 A：通过 CI 自动构建（推荐）

把代码合并到 `main` 分支并 push，GitHub Actions 会自动构建所有镜像（包括新增的 patroni）：

```bash
# 在你的本地开发机执行
git add -A
git commit -m "feat: add Patroni HA for PostgreSQL auto-failover"
git push origin main
```

然后去 GitHub 仓库的 **Actions** 页面，等所有构建完成（绿色勾）。

### 方式 B：手动构建

如果不想走 CI，在能访问 Docker Registry 的机器上：

```bash
# 在项目根目录执行
cd /path/to/ho

# 构建 Patroni 镜像（把 <你的用户名> 替换为你的 GitHub 用户名或组织名）
docker build -t ghcr.io/<你的用户名>/ecom-patroni:latest -f infra/patroni/Dockerfile .

# 推送到 Registry
docker push ghcr.io/<你的用户名>/ecom-patroni:latest
```

## 步骤 6：上传配置文件到服务器

> **在哪执行：** 你的本地开发机

把新增和修改的配置文件传到 Swarm Manager 节点：

```bash
# 把整个 infra 目录上传到服务器的 /opt/ecom/
scp -r infra/ <用户名>@<S1的IP>:/opt/ecom/
```

**确认成功：** SSH 到服务器检查文件存在：

```bash
ssh <用户名>@<S1的IP>

# 检查新文件
ls -la /opt/ecom/infra/patroni/
ls -la /opt/ecom/infra/haproxy/
```

应该看到：

```
/opt/ecom/infra/patroni/
  Dockerfile
  entrypoint.sh
  patroni.yml
  post-init.sh

/opt/ecom/infra/haproxy/
  haproxy.cfg
```

## 步骤 7：部署新 Stack（核心操作）

> **在哪执行：** Swarm Manager 节点（SSH 到 S1）
>
> **预计停机：** 2-5 分钟（旧 PG 停止 → Patroni 接管 → HAProxy 开始路由）

```bash
cd /opt/ecom/infra/swarm

# 登录镜像仓库（Worker 节点拉镜像需要凭证）
echo "<你的GHCR PAT>" | docker login ghcr.io -u <GitHub用户名> --password-stdin

# 部署！（把 <你的用户名> 替换为实际值）
REGISTRY=ghcr.io/<你的用户名> TAG=latest \
  docker stack deploy -c docker-stack.yml --with-registry-auth ecom
```

`docker stack deploy` 是**幂等**的：
- 新服务（etcd、patroni、haproxy）→ 创建
- 已删除的服务（postgres-primary、postgres-replica）→ 移除
- 修改过的服务（5 个微服务的 DATABASE_URL 变了）→ 滚动更新

部署命令发出后，等待 1-2 分钟，观察进度：

```bash
# 实时查看所有服务状态（每 2 秒刷新）
watch -n2 'docker service ls --filter "name=ecom_"'
```

等所有服务的 `REPLICAS` 列都显示 `1/1`（或 `2/2` for haproxy 和 sentinel）再继续。

如果某个服务一直是 `0/1`，查看原因：

```bash
# 查看失败的任务
docker service ps ecom_patroni-1 --no-trunc

# 查看日志
docker service logs ecom_patroni-1
```

## 步骤 8：验证集群状态

> **在哪执行：** Manager 节点

### 8.1 检查 etcd 集群

```bash
docker exec $(docker ps -q -f name=ecom_etcd-1) etcdctl member list
```

应看到 3 个成员。

### 8.2 检查 Patroni 集群（最重要）

```bash
docker exec $(docker ps -q -f name=ecom_patroni-1) patronictl list
```

期望输出：

```
+----------+------------+---------+---------+----+-----------+
| Member   | Host       | Role    | State   | TL | Lag in MB |
+----------+------------+---------+---------+----+-----------+
| patroni-1| patroni-1  | Leader  | running |  1 |           |
| patroni-2| patroni-2  | Replica | running |  1 |         0 |
+----------+------------+---------+---------+----+-----------+
```

关键确认：
- 一个 `Leader` + 一个 `Replica`
- 两个都是 `running`
- `Lag in MB` 为 0 或接近 0

### 8.3 检查 HAProxy 路由

```bash
# 从应用网络内部测试连接
docker exec $(docker ps -q -f name=ecom_patroni-1) \
  psql -h haproxy -p 5432 -U postgres -d ecommerce -c "SELECT 'OK';"
```

能返回 `OK` 说明 HAProxy 路由正常。

### 8.4 检查应用健康

```bash
curl -X POST https://api.find345.site/health
```

返回 `"success": true` 就是全部正常了。

## 步骤 9：测试故障转移（可选但强烈推荐）

> 模拟 S1 宕机，验证自动切换是否真的生效。

### 9.1 确认当前状态

```bash
docker exec $(docker ps -q -f name=ecom_patroni-1) patronictl list
```

确认 patroni-1 是 `Leader`。

### 9.2 模拟宕机

```bash
# 把 patroni-1 缩容为 0（模拟 S1 宕机）
docker service scale ecom_patroni-1=0
```

### 9.3 等待自动切换（约 30 秒）

```bash
# 等 35 秒让 Patroni 完成选举
sleep 35

# 检查 patroni-2 是否自动提升为 Leader
docker exec $(docker ps -q -f name=ecom_patroni-2) patronictl list
```

期望看到 patroni-2 变成了 `Leader`：

```
+----------+------------+---------+---------+----+-----------+
| Member   | Host       | Role    | State   | TL | Lag in MB |
+----------+------------+---------+---------+----+-----------+
| patroni-2| patroni-2  | Leader  | running |  2 |           |
+----------+------------+---------+---------+----+-----------+
```

### 9.4 验证应用仍然可用

```bash
curl -X POST https://api.find345.site/health
```

应返回成功。

### 9.5 恢复 patroni-1

```bash
# 恢复 patroni-1（它会自动作为 Replica 加入，不会抢回 Leader）
docker service scale ecom_patroni-1=1

# 等 30 秒后再次检查
sleep 30
docker exec $(docker ps -q -f name=ecom_patroni-2) patronictl list
```

最终状态：

```
+----------+------------+---------+---------+----+-----------+
| Member   | Host       | Role    | State   | TL | Lag in MB |
+----------+------------+---------+---------+----+-----------+
| patroni-1| patroni-1  | Replica | running |  2 |         0 |
| patroni-2| patroni-2  | Leader  | running |  2 |           |
+----------+------------+---------+---------+----+-----------+
```

> patroni-1 恢复后变成 **Replica** 而非 Leader，这是正常行为。

## 如果出问题了

### 回滚方案

如果部署后应用无法访问，最快的恢复方式是用旧的 stack 文件重新部署：

```bash
# 前提：你保留了旧的 docker-stack.yml
docker stack deploy -c docker-stack.yml.bak --with-registry-auth ecom
```

> 建议在步骤 7 之前先备份旧文件：`cp docker-stack.yml docker-stack.yml.bak`

### 常见问题

| 现象 | 可能原因 | 解决 |
|------|----------|------|
| patroni 启动失败 | 镜像没推送到 Registry | `docker service logs ecom_patroni-1` 查看错误信息 |
| etcd 集群起不来 | 3 个 etcd 需要同时启动才能选举 | 等 1-2 分钟让它们互相发现 |
| 应用连不上数据库 | HAProxy 还没检测到 Leader | 等 30 秒让健康检查收敛 |
| `patronictl list` 显示 Lag 很大 | Replica 正在从 Leader 同步数据 | 等同步完成即可 |
| `replicator` 认证失败 | Secret 密码和步骤 2 创建的不一致 | 检查是否 `echo -n` 漏了 `-n` |
