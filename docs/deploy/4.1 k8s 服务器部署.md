# Kubernetes 部署指南

> 从 Docker Swarm 迁移到 Kubernetes（kubeadm 自建集群 + Helm Chart）

## 前置条件

- 5 台 Ubuntu 22.04+ 服务器（S1-S5），最低配置 2C4G
- 所有节点之间网络互通
- 一个可用的 VIP 地址（用于 kube-vip Control Plane HA）
- 一个私有镜像仓库（如 Harbor）
- 域名 `api.find345.site` 解析到 S3 的公网 IP

## 节点规划

| 节点 | kubeadm 角色 | K8s 标签 | 运行的工作负载 |
|------|-------------|---------|--------------|
| S1 | Control Plane | `role=data` | CloudNativePG Pod 1, Redis Pod |
| S2 | Control Plane | `role=data` | CloudNativePG Pod 2, Redis Pod |
| S3 | Control Plane | `role=ingress` | Nginx Ingress Controller |
| S4 | Worker | `role=app` | 微服务 Pods |
| S5 | Worker | `role=app` | 微服务 Pods |

## 架构图

```
                    Internet
                       ↓
              DNS → S3 IP (不变)
                       ↓
            Nginx Ingress Controller (hostNetwork on S3)
            cert-manager 自动管理 TLS 证书
                       ↓
                K8s Service (ClusterIP)
                       ↓
              API Gateway Deployment (×2)
                       ↓
         ┌──────┬──────┬──────┬──────┐
       user   product  cart   order    (Deployment ×2 each)
         └──────┴──────┴──────┴──────┘
                       ↓
           ┌───────────┴───────────┐
     CloudNativePG              Redis Operator
   (替代 Patroni+HAProxy+etcd)  (替代手动主从+Sentinel)
     S1/S2 nodeSelector          S1/S2 nodeSelector
```

## 第一步：搭建 K8s 集群

### 1.1 所有节点安装前置依赖（S1-S5）

```bash
# 在每台服务器上执行
sudo bash infra/k8s/cluster-setup/01-prerequisites.sh
```

此脚本会：
- 禁用 swap
- 加载 `overlay` / `br_netfilter` 内核模块
- 设置 `net.ipv4.ip_forward = 1` 等 sysctl 参数
- 安装 containerd（启用 SystemdCgroup）
- 安装 kubeadm / kubelet / kubectl v1.29 并锁定版本

### 1.2 初始化 Control Plane（S1）

```bash
# 设置必要的环境变量
export KUBE_VIP_ADDRESS=<你的VIP地址>
export S1_IP=<S1的内网IP>
export KUBE_VIP_INTERFACE=eth0  # 根据实际网卡名修改

# 修改 kubeadm-config.yaml 中的 S2_IP、S3_IP（certSANs 部分）
vim infra/k8s/cluster-setup/kubeadm-config.yaml

# 初始化
sudo -E bash infra/k8s/cluster-setup/02-init-control-plane.sh
```

初始化完成后，记录输出的 join token、discovery hash 和 certificate key。

### 1.3 加入 Control Plane（S2、S3）

```bash
export KUBE_VIP_ADDRESS=<VIP地址>
export JOIN_TOKEN=<从S1获取>
export DISCOVERY_HASH=<从S1获取>
export CERTIFICATE_KEY=<从S1获取>

sudo -E bash infra/k8s/cluster-setup/03-join-control-plane.sh
```

### 1.4 加入 Worker（S4、S5）

```bash
export KUBE_VIP_ADDRESS=<VIP地址>
export JOIN_TOKEN=<从S1获取>
export DISCOVERY_HASH=<从S1获取>

sudo -E bash infra/k8s/cluster-setup/04-join-workers.sh
```

### 1.5 安装 CNI（S1 执行）

```bash
bash infra/k8s/cluster-setup/05-install-cni.sh
```

验证所有节点 Ready：

```bash
kubectl get nodes
# 应该看到 5 个节点，状态均为 Ready
```

### 1.6 安装 Operator 和组件（S1 执行）

```bash
export S1_NODE=<S1节点名>
export S2_NODE=<S2节点名>
export S3_NODE=<S3节点名>
export S4_NODE=<S4节点名>
export S5_NODE=<S5节点名>

bash infra/k8s/cluster-setup/06-install-operators.sh
```

此脚本安装：
- **local-path-provisioner** — 本地持久化存储（设为默认 StorageClass）
- **CloudNativePG Operator** — 替代 etcd + Patroni + HAProxy
- **Redis Operator (OT-Container-Kit)** — 替代手动 Redis 主从 + Sentinel
- **Nginx Ingress Controller** — 替代 Caddy（DaemonSet + hostNetwork 绑定 S3）
- **cert-manager** — 自动 TLS 证书管理

同时设置节点标签和 taint。

## 第二步：部署应用

### 2.1 初始化 & 创建 Secrets

```bash
cd infra/k8s
export REGISTRY=<你的镜像仓库地址>

./deploy.sh setup
```

按提示输入各密码/密钥。

### 2.2 构建镜像

```bash
export TAG=$(git rev-parse --short HEAD)
./deploy.sh build
```

### 2.3 Helm 部署

```bash
./deploy.sh deploy
```

### 2.4 验证

```bash
./deploy.sh status
```

检查：
- 所有 Pod 为 Running 状态
- CloudNativePG Cluster 为 Ready
- Redis Replication 为 Ready
- Ingress 已分配规则
- Certificate 已签发

```bash
# 端到端验证
curl -v https://api.find345.site/health
```

## 第三步：数据迁移（从 Swarm 切换）

### 最小停机时间方案

K8s 集群搭建可以在 Swarm 运行的同时进行。数据迁移步骤：

```bash
# 1. 停 Swarm 应用层（数据层仍然运行）
docker service scale ecom_api-gateway=0 ecom_user-service=0 \
  ecom_product-service=0 ecom_cart-service=0 ecom_order-service=0

# 2. 导出 PostgreSQL 数据
docker exec $(docker ps -q -f name=ecom_patroni-1) \
  pg_dump -U postgres -d ecommerce -Fc > ecommerce.dump

# 3. 导入到 CloudNativePG
# 获取 PG Pod 名
PG_POD=$(kubectl get pods -n ecom -l cnpg.io/cluster=ecom-pg -o jsonpath='{.items[0].metadata.name}')
# 复制 dump 文件到 Pod
kubectl cp ecommerce.dump ecom/${PG_POD}:/tmp/ecommerce.dump
# 执行恢复
kubectl exec -n ecom ${PG_POD} -- pg_restore -U postgres -d ecommerce /tmp/ecommerce.dump

# 4. Redis 数据（可选，购物车数据可以重建）
# 如果需要迁移：导出 RDB → 挂载到新 Redis Pod

# 5. 部署应用层
./deploy.sh deploy

# 6. 验证
curl -v https://api.find345.site/health

# 7. DNS 不需要改（Ingress 绑定 S3 同一 IP）

# 8. 验证期后拆除 Swarm
docker stack rm ecom
```

### 回滚方案

```bash
# K8s 部署失败 → 卸载
helm uninstall ecom -n ecom

# 恢复 Swarm 应用层
docker service scale ecom_api-gateway=1 ecom_user-service=1 \
  ecom_product-service=1 ecom_cart-service=1 ecom_order-service=1
```

## 日常运维

### 更新部署

```bash
# 构建新镜像
export TAG=$(git rev-parse --short HEAD)
./deploy.sh build

# 滚动更新
./deploy.sh deploy
```

### 数据库迁移

```bash
./deploy.sh migrate
```

### 回滚

```bash
# 查看历史版本
helm history ecom -n ecom

# 回滚到指定版本
./deploy.sh rollback <revision>
```

### 扩缩容

```bash
# 扩展 API Gateway 到 3 副本
helm upgrade ecom infra/k8s/ecom-chart -n ecom --set apiGateway.replicas=3

# 或修改 values.yaml 后
helm upgrade ecom infra/k8s/ecom-chart -n ecom
```

### 查看日志

```bash
# 某个服务的日志
kubectl logs -f -l app.kubernetes.io/name=api-gateway -n ecom

# 所有服务的日志
kubectl logs -f -l app.kubernetes.io/part-of=ecom -n ecom
```

### 故障排查

```bash
# Pod 状态
kubectl describe pod <pod-name> -n ecom

# PG 集群状态
kubectl get cluster -n ecom -o yaml

# Redis 状态
kubectl get redisreplication -n ecom -o yaml

# 证书状态
kubectl describe certificate -n ecom

# Ingress 状态
kubectl describe ingress -n ecom
```

## 与 Swarm 对比

| 维度 | Docker Swarm | Kubernetes |
|------|-------------|------------|
| 入口层 | Caddy 单点 | Nginx Ingress + cert-manager（可扩展 HA） |
| PG 高可用 | etcd + Patroni + HAProxy（7 个容器） | CloudNativePG（1 个 Operator，自动管理） |
| Redis 高可用 | 主从 + Sentinel（5 个容器） | Redis Operator（1 个 Operator，自动管理） |
| 滚动更新 | `start-first` 策略 | `maxUnavailable: 0, maxSurge: 1` |
| Secret 管理 | Docker Secret（文件挂载） | K8s Secret（环境变量注入） |
| 扩缩容 | `docker service scale` | `kubectl scale` / `helm --set replicas` |
| 自愈 | 容器重启 | Pod 自动重调度到其他节点 |
| 回滚 | 手动 | `helm rollback` 一键回滚 |
| 证书管理 | Caddy 自动 | cert-manager 自动 |

## 废弃组件（保留但不再使用）

迁移到 K8s 后，以下 Swarm 组件不再需要：

- `infra/swarm/docker-stack.yml` → Helm Chart
- `infra/swarm/Caddyfile.prod` → Nginx Ingress + cert-manager
- `infra/swarm/deploy.sh` → `infra/k8s/deploy.sh`
- `infra/swarm/redis-sentinel.conf` → Redis Operator
- `infra/patroni/` (全部) → CloudNativePG
- `infra/haproxy/haproxy.cfg` → CloudNativePG 自动创建 Service
